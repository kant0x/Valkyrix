# Phase 02.2: In-game menu and overlay UX - Context

**Gathered:** 2026-03-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Refresh the in-match player command surface without changing the game into a different genre or adding unrelated meta screens. This phase covers the always-visible in-game command panel, the relationship between HUD and command controls, and the role of ESC as a compact system overlay while a match is running.

</domain>

<decisions>
## Implementation Decisions

### In-match command surface
- Phase 02.2 is not "just an ESC menu". It is primarily an in-game command panel that stays visible during battle.
- The preferred format is a bottom command bar, not a side dock and not a hidden-on-demand control layer.
- The command bar should read as the main battle workspace for the player during a live match.
- The existing HUD should evolve into this command layer instead of staying a tiny passive status strip plus temporary floating buttons.

### Core interaction scope
- First-version command panel should focus on `Build + Train + Resources` as the war loop framing.
- Real tower actions should be part of this phase because tower placement already exists in the game.
- Resource visibility should be integrated directly into the command panel rather than treated as a separate tiny HUD stat.
- Unit training is desired in the panel design, but the current gameplay code does not yet expose a complete player-facing training flow; the panel should therefore be designed with staged unit-control slots rather than forcing a fake fully-complete command system.
- This phase should replace the temporary free-floating tower buttons in the game screen with a proper integrated control surface.

### Match behavior
- The command panel should be always visible during battle.
- Phase 2's no-pause rule remains intact: opening ESC must not halt simulation.
- If visual quieting is needed, it should be handled through styling and focus, not by pausing the match.
- The in-match layer should feel tactical and operational, not like a separate menu screen sitting on top of the game.

### ESC and overlay hierarchy
- ESC remains a compact system menu, not the main place for war controls.
- ESC should own short system actions such as resume, settings/music, and exit flow.
- Battle actions should live in the bottom command bar, not inside the ESC overlay.
- The result overlay (victory/defeat) belongs to the same visual family as the command layer and ESC, but remains a distinct end-of-match state.

### Visual direction
- The in-match panel should align with the new Valkyrix direction but remain more tactical and compact than the heavy main-menu frame language.
- The command bar should feel like a battle console: readable, sturdy, rune-touched, and clearly divided into actionable regions.
- Visual density should support quick use in combat; ornamental excess should not crowd the map.
- The layout should make room for current real controls first, then for future staged unit actions, rather than pretending all systems already exist.

### Claude's Discretion
- Exact sub-layout inside the bottom bar
- Whether command sections read as cards, trays, or segmented rails
- Exact iconography, typography, and micro-motion
- How strongly to restyle victory/defeat presentation relative to HUD and ESC

</decisions>

<specifics>
## Specific Ideas

- The user clarified that this phase is "not exactly a menu" but more of an in-game war panel where towers, warriors, and farming-related actions should live.
- The user's preferred structure is a persistent bottom battle bar.
- The user wants the first version to center on practical wartime controls rather than a decorative popup.
- Existing temporary tower controls in `main.ts` are a clear replacement target for this phase.
- The desired reading is: battle control surface first, system menu second.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/screens/HudOverlay.ts`: already owns persistent in-match UI mounting and live `update(...)` calls for wave, health, and resources.
- `src/screens/EscMenuOverlay.ts`: already handles keybinding, open/close behavior, and exit-to-menu flow without pausing gameplay.
- `src/game/BuildingSystem.ts`: already has real building placement and resource deduction for `attack` and `buff` towers.
- `src/main.ts`: already routes game state into HUD updates and contains the temporary tower control logic that this phase should absorb.

### Established Patterns
- Screen/UI code is still vanilla TypeScript + DOM modules with local style injection.
- Overlays are mounted by the game screen, not by `ScreenManager`, so the command layer should probably stay in that same runtime-owned pattern.
- Match simulation continues while overlays are open; this is an established behavioral contract from earlier phases.

### Integration Points
- Temporary buttons `btn-attack-tower` and `btn-buff-tower` in `src/main.ts` are the clearest first integration point for the new command panel.
- `selectedTowerType` and the canvas click placement flow already provide a real interaction path the new panel can drive.
- `HudOverlay.update(...)` already receives `wave`, `health`, `citadelMaxHp`, and `resources`, so the command panel can reuse these values instead of inventing a new game-state feed.
- Victory/defeat currently renders through `HudOverlay.showWinLossOverlay(...)`, making it a natural follow-up surface for visual alignment in this phase.

</code_context>
