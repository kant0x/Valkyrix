  <action>
    In `GameScreen.mount(container)`, remove the direct creation of `btn-attack-tower` and `btn-buff-tower`.

    Keep the existing module-level `selectedTowerType: 'attack' | 'buff' | null = null`, but route all state changes through the new HUD API. Use these exact callback mappings:
    - `attack`: toggles `selectedTowerType` between `'attack'` and `null`
    - `buff`: toggles `selectedTowerType` between `'buff'` and `null`
    - `stagedUnitA`: calls `hud.setCommandMessage('Command Slot Locked - This war control is staged for a later step. Hold the line with towers and resources for now.')`
    - `stagedUnitB`: same locked-slot message

    After mounting the HUD, call `hud.setCommandCallbacks({...})` with those exact actions.

    Whenever selection changes, call `hud.setBuildSelection(selectedTowerType)`.

    During live HUD updates from `gameState`, also call:
    - `hud.setActionAvailability({ attack: gameState.resources >= 50, buff: gameState.resources >= 40 })`
    - `hud.update({ wave: gameState.waveNumber, health: gameState.citadelHp, citadelMaxHp: gameState.citadelMaxHp, resources: gameState.resources, armedAction: selectedTowerType })`

    In the existing canvas placement handler:
    - on success set `hud.setCommandMessage('Attack tower armed and placed.')` or `hud.setCommandMessage('Buff tower armed and placed.')`
    - on failure set `hud.setCommandMessage('Action unavailable - not enough resources or invalid placement.')`
    - after a successful placement, clear the current selection by setting `selectedTowerType = null` and calling `hud.setBuildSelection(null)`

    In `GameScreen.unmount()`, remove only the canvas click listener and state references. Do not attempt to remove the deleted floating buttons anymore.
  </action>
