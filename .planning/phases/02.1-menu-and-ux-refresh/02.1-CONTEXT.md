# Phase 02.1: Menu and UX refresh - Context

**Gathered:** 2026-03-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Refresh the Phase 2 player-facing UX without changing its core product boundary: wallet splash, main menu, screen transitions, and Phase 2 overlays should feel more intentional and cohesive, while keeping the existing screen architecture, wallet requirement, no-reload routing, and Play-to-game flow intact.

</domain>

<decisions>
## Implementation Decisions

### Visual Direction
- The redesign should blend all three moods the user pointed at: dark fantasy atmosphere, tactical clarity, and light web3 accents.
- Dark fantasy is the primary layer: richer mood, stronger world feel, less "plain placeholder UI".
- Tactical clarity is the structural layer: screens must stay readable and decisive, not ornamental clutter.
- Web3 styling is secondary and should appear in details, status cues, and wallet language rather than dominate the whole interface.
- The main menu should draw from a heavy nordic runic frame aesthetic: icy blue runes, carved metal/stone feeling, and a central framed presentation area.
- The visual exploration should not lock to one composition immediately: UI-spec should produce 2-3 menu concepts inside this art direction before implementation picks one.

### Main Menu Structure
- The menu should stay Play-first.
- `Play` remains the primary CTA and should visually dominate the screen.
- Secondary actions stay quieter than Play; the menu should not turn into a noisy dashboard.
- The phase remains within the original Phase 2 scope: no map select, no full lobby system, no new feature tree.
- The preferred composition is Hero + status zone: a strong central hero area supported by a separate information/status region rather than a pure single-column splash or a full hub layout.
- The menu should feel more designed and inhabited than the current placeholder approach; it should not read as "just a frame with buttons."
- Visible secondary actions should be `Leaderboard` and `Stats`, both clearly secondary to `Play`.
- The status zone should prioritize wallet identity and readiness to enter the game.
- Menu copy should lean epic, but remain concise and readable.

### Wallet Experience
- Wallet remains mandatory before entering the game.
- The wallet splash should become a softer onboarding step: explain why wallet connection is needed and reduce perceived friction.
- The screen should feel less like a hard blocker and more like an intentional entry ritual into the game.
- Wallet status can be surfaced more clearly in the menu after connection, as long as this stays within Phase 2 presentation and does not become a new blockchain feature.
- Wallet-screen tone should feel like a ritualized entry into the world, not a dry auth form.
- Wallet explanation should stay short and direct: brief reasons, no long technical wall of text.
- Phantom and Backpack should be presented as two equal, deliberate choices rather than plain utility buttons.
- If the wallet is already connected and has not been disconnected, the app should skip the wallet screen and open the main menu directly.

### Routing and Interaction Constraints
- Existing `ScreenManager` state-machine routing stays in place.
- Navigation remains DOM-based with no page reload.
- `Play` still transitions into the game session flow.
- ESC overlay remains a real-time overlay and must not pause gameplay.
- Quick wallet management actions should not become a major menu concern in this phase; the menu stays game-first rather than wallet-control-first.

### Identity and Readiness Presentation
- The connected wallet should be shown in a visible status block, not hidden away as a tiny detail.
- The wallet block should be relatively informative rather than minimal, but still readable and integrated into the scene.
- Ready-state messaging should feel like battle readiness rather than a sterile "connected" system message.
- Wallet labels should be game-flavored but still easy to understand at a glance.

### Motion and Menu Scene
- Transitions between wallet -> menu -> game should feel like a soft ritual: fades, glow, and atmosphere rather than abrupt screen swaps.
- The heavy runic frame language should apply strongly to the main menu only.
- Wallet-screen can share the family resemblance, but does not need the same massive frame treatment.
- HUD and ESC overlay should only gain light stylistic alignment with the refreshed menu; they should not inherit the full heavy frame treatment.
- The center of the main menu should still be UI-led: background atmosphere supports the interface rather than overpowering it.
- The interior menu scene should be moderately alive: subtle motion, glow, particles, or parallax are welcome, but the screen should not become a hyperactive cinematic.
- Blue rune glow should be expressive and signature-making, but controlled enough to stay clean and legible.

### Claude's Discretion
- Exact typography choices
- Motion and transition details between wallet/menu/game
- The amount of decorative worldbuilding copy vs terse system language
- How far to visually align HUD and ESC overlay with the new menu language
- Which of the 2-3 explored menu concepts becomes the implementation favorite after UI-spec

</decisions>

<specifics>
## Specific Ideas

- The user wants to "go back to Phase 2" and rework the menu/UX rather than treat the current UI as final.
- The desired outcome is not just a cosmetic tweak to one button, but a more coherent Phase 2 presentation pass.
- "All" visual directions were selected; interpret this as a blended direction:
  dark-fantasy base, tactical UI organization, subtle Solana/MagicBlock flavor.
- The user provided a strong visual reference for the main menu: a heavy nordic runic frame with icy blue glow, skull/warrior ornamentation, and a large central empty area intended to hold the hero UI.
- That reference should influence the main menu specifically, not automatically the entire UI system.
- The desired center usage is a hero UI block inside the frame, not a mostly empty art scene with controls scattered around the edges.
- `Map selection` came up as a future desire, but not as scope for Phase 02.1.
- The game fiction now explicitly supports this menu direction: the player is the last Valkyrix, defending the Citadel and preparing a counterattack against four Voidspawn portals.
- Menu and wallet copy can directly reference the Citadel, the Code of the Ancestors, and the portal-war setup instead of relying on placeholder fantasy flavor.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/screens/ScreenManager.ts`: simple state-machine router already matches the desired no-reload flow and should not need architectural replacement.
- `src/main.ts`: central integration point already mounts wallet, menu, game, HUD, and ESC layers through the existing screen flow.
- Existing screen modules already use local `ensureStyle()` CSS injection, so a visual refresh can happen without introducing a framework.

### Established Patterns
- Vanilla TypeScript + DOM modules remain the implementation style.
- Screen modules follow `mount(container)` / `unmount()` and should keep that contract.
- Overlay modules (`EscMenuOverlay`, `HudOverlay`) are separate from `ScreenManager` and can be visually refreshed without changing routing rules.

### Integration Points
- `WalletSplashScreen` owns the mandatory wallet gate before menu access.
- `MainMenuScreen` owns the `Play` connection step and is the key surface for the redesign.
- `src/main.ts` is where any updated transitions or screen composition will be felt end-to-end.
- HUD and ESC overlay should be checked for visual consistency with the refreshed menu language.
- Wallet reconnect behavior will likely require coordination between wallet service state and initial screen selection in `src/main.ts`.

</code_context>
